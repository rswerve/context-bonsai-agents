# Context Bonsai Meta-Loop: Direction

This document states the end goal for the Context Bonsai meta-loop, the next step toward it, and the provisional steps after that. It is itself an iteration artifact: the process that produced it — examine the loop's current state, absorb what recent cycles taught, restate direction — is expected to run again, and a later iteration may overwrite the "Next Step" and "Provisional Future Steps" sections wholesale. Only the End Goal section should survive iterations largely intact; treat edits to it as scope changes requiring the project owner's approval.

## End Goal

A self-updating Context Bonsai that works for every supported harness.

Concretely, the system maintains three levels of artifact, each able to regenerate the one below it:

1. **Behavior contract** (`docs/context-bonsai-agent-spec.md`) — what Context Bonsai does, harness-independent. Changes only when the product changes.
2. **Per-harness update loop** — the routine procedure that carries a port onto each new harness release. Two upstream *shapes* exist today: for *git-fork* harnesses, rebase the patch chain and revalidate; for *closed-artifact* harnesses (Claude Code), freeze the new npm artifact, re-verify semantic anchors, and revalidate. This level encodes structural assumptions about the harness (package layout, seam locations, validation commands) that hold across the majority of releases.
3. **Per-release cycle** — one execution of level 2 against one new release, ending in the e2e release gate defined by `docs/agent-specs/context-bonsai-e2e-spec.md`.

The meta-loop is the machinery that generates and regenerates level 2. It must handle three entry conditions with one pipeline, differing only in how much prior work is still trustworthy:

- **Routine release** — level 2's assumptions hold. Generate and execute a cycle plan. This is the cheap path and the common case.
- **Structural break** — a harness release invalidates level 2's assumptions. The signals already exist as fail-closed behaviors: mass classification of commits into the hard-blocking `manual_review` bucket (the cycle generator's fallback for indeterminate evidence), anchor discovery scoring below the `minScore`/`minMargin` thresholds everywhere (the behavior spec's fail-closed anchor-ambiguity rule), or a Protocol A failure (the e2e spec's secret-prune oracle) on a clean build. The failure is the escalation signal: re-derive the level-2 assumptions from the behavior contract plus fresh exploration of the new harness architecture, demoting prior bindings to untrusted priors that must be re-verified, then re-enter the routine path.
- **New harness** — no prior work exists. Run the same derivation from nothing: capability discovery, integration-posture selection (per the behavior spec's change-minimization rule), per-harness spec, implementation, e2e binding, and emission of that harness's level-2 update loop as a first-class artifact.

"Self-updating" means the routine path runs with no human decision-making, and the other two paths run agentically with human review at the existing reviewer+judge gates. Detection of which path applies is itself automatic — the routine path fails closed with a structural reason code rather than grinding forward or silently succeeding, and a path-selection dispatcher routes the code without a human interpreting the STOP.

**Executor tiering.** The determinism machinery (frozen inputs, fixed precedence, machine-checkable gates) is calibrated scaffolding, not architecture: each such rule moves a judgment call out of the executor and into the artifact, and that is precisely what lets weaker or cheaper models run the loops. The assignment is per-harness, not global — three owner decisions set the current shape (the branch-1 ratification 2026-07-03, which up-tiered plan generation to the owner class while leaving execution on the weak tier; the Claude Code calibration-target direction 2026-07-03; the executor-model change 2026-07-04): **cycle-plan generation and validation always run on the owner (Fable-class) tier; execution of the approved plan and the routine's own §1.16 maintenance (section numbers throughout cite `docs/agent-specs/forward-port-spec.md` unless a sentence names another spec, whose own numbering then applies; §1.16 absorbs cycle friction back into the routine's instructions) run on the cheapest tier that has proven reliable for that harness** — for OpenCode, a process orchestrator (`.llm-conductor/ORCHESTRATOR_AGENT.md` pattern) on `openai/gpt-5.3-codex-spark`; for Claude Code, an Opus 4.8 subagent at low effort. (The OpenCode pilot's first four runs executed on GPT-5.5 via OpenCode's own Zen `opencode/*` subscription-hosted provider; a quota mis-route exposed that wiring, the owner forbade Zen for pilot runs and moved execution to his OpenAI subscription, which offers no GPT-5.5 — hence `gpt-5.3-codex-spark` from run 5b onward.) Two layers of scaffolding are now distinguished by evidence: the deterministic gates catch false *outcomes* (nothing false has ever sealed or pushed), and the §1.18 run-continuity mechanisms (intent log, reconciliation, report-before-stop) keep a weak executor's *account of its own actions* truthful across process boundaries — a failure layer the gates cannot reach, observed as confident confabulation on `gpt-5.3-codex-spark` before §1.18 existed. Executors below the owner tier get both layers; relaxing either may be explored only in tiers executed by the owner-class model. The loop above all of this — developing the loop system, governed by this document's Iteration Rule — runs on the owner-class model with the project owner.

## Provisional Future Steps

1. **Bind the remaining unbound harnesses** by repeating the Next Step's Stage 5–6 run until all four are covered. Each binding extends the detector's coverage and creates a routine path where none exists; each run's lessons fold into the pipeline spec before the next.

Detection, generation, execution, escalation routing, and local landing are each individually built or ruled routine; what remains is chaining them so a routine release consumes zero owner decisions between signal and sealed local landing (outward actions stay owner-gated by standing order).
